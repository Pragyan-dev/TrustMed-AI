## Expectations

- SOAP note generation should use Gemini 3 Flash Preview through OpenRouter by
  default.
- The default SOAP model is controlled by `SOAP_MODEL` and currently resolves to
  `google/gemini-3-flash-preview`.
- If `SOAP_MODEL` is set in the environment, that value should override the
  default.
- The SOAP endpoint should produce structured SOAP JSON when the model returns
  valid JSON.
- If the SOAP LLM returns empty, prose, malformed JSON, or otherwise unparseable
  content, the backend should continue returning a structured note through the
  demo fallback instead of failing the request.
- Streaming chat should stop reading the SSE stream when a `done` or `error`
  event is received.
- Long-running synthesis should emit progress heartbeat events instead of
  appearing stalled while waiting for model tokens.
- Safety critic calls should use bounded timeout, retry, and token settings so
  they do not stall chat indefinitely.
- Vertex MedGemma vision calls should prefer ADC/service-account credentials,
  then fall back to the active `gcloud` user token when appropriate.
- All changes described below were pushed. There should be no required local-only
  generated artifacts for another developer to pick up this state.

### Orchestration And SOAP Logic: `src/trustmed_brain.py`

Commit `72b2465` made several behavior changes:

- Imported the new runtime config values listed above.
- Changed the default safety critic model from `google/gemma-3n-e4b-it:free` to
  `minimax/minimax-m2.7`.
- Changed `SAFETY_CRITIC_FALLBACKS` from a hardcoded list to an env-driven comma
  separated list.
- Updated `_invoke_with_retry` so a rate-limit error on the last retry is raised
  instead of sleeping/retrying past the configured retry budget.
- Added `_synthesis_stream_with_heartbeats(...)`, which wraps token generation
  and emits heartbeat progress while waiting for the next token.
- Updated streaming synthesis to use:
  - `SYNTHESIS_MAX_TOKENS`
  - `SYNTHESIS_REQUEST_TIMEOUT_SECONDS`
  - `SYNTHESIS_FIRST_TOKEN_TIMEOUT_SECONDS`
  - `SYNTHESIS_IDLE_TOKEN_TIMEOUT_SECONDS`
  - `SYNTHESIS_DIRECT_FALLBACK_TIMEOUT_SECONDS`
- Updated direct fallback synthesis to use `SYNTHESIS_MAX_TOKENS` and
  `SYNTHESIS_DIRECT_FALLBACK_TIMEOUT_SECONDS`.
- Updated safety critic calls to use:
  - `SAFETY_CRITIC_MAX_TOKENS`
  - `SAFETY_CRITIC_TIMEOUT_SECONDS`
  - `SAFETY_CRITIC_MAX_RETRIES`
- Kept the existing safety-mode gate, so the LLM critic still honors
  `TRUSTMED_SAFETY_MODE` and the pipeline plan.
- Expanded `SOAP_NOTE_PROMPT` so the SOAP model receives patient context,
  current medications, drug-safety alerts, imaging/lab findings, and conversation
  history as separate sections.
- Added explicit SOAP instructions:
  - synthesize patient context instead of just quoting it
  - use current medications as background only
  - put only newly recommended drugs in `plan.medications`
  - cite drug-safety concerns where relevant
  - return only valid JSON
- Added `_build_demo_soap_fallback(...)` for pneumonia, Mycoplasma pneumonia,
  pleural effusion, atelectasis, and proximal humerus fracture scenarios.
- Added `_is_soap_too_sparse(...)` to detect essentially empty SOAP responses.
- Updated `generate_soap_note(...)` to accept:
  - `history`
  - `patient_context`
  - `vision_context`
  - `current_medications`
  - `drug_alerts`
- Changed SOAP failure behavior from returning a sparse raw-text wrapper to using
  a clinically coherent fallback when JSON parsing fails or the model output is
  too sparse.

Commit `c03928a` changed SOAP model selection:

- Imported `SOAP_MODEL` from runtime config.
- Replaced the hardcoded SOAP model list
  `["nvidia/nemotron-3-nano-30b-a3b:free"]` with `[SOAP_MODEL]`.

Expected SOAP path now:

1. Build a structured SOAP prompt.
2. Call OpenRouter with `SOAP_MODEL`.
3. Parse the model response as JSON.
4. If parsing fails, log:
   `LLM produced unparseable output, using demo fallback`
5. Return fallback structured SOAP JSON instead of failing the endpoint.

### FastAPI Backend: `api/main.py`

Commit `72b2465` updated backend endpoint behavior:

- Imported `check_drug_interactions` from `src.trustmed_brain`.
- Updated streaming chat internals so `_stream_chat(...)` returns immediately
  after sending a `done` event.
- Updated streaming chat error handling so it returns immediately after sending
  an `error` event.
- Expanded `/soap-note` endpoint context assembly:
  - loads patient context when `patient_id` is provided
  - loads structured patient data via `get_patient_data_json`
  - extracts current medication names from the patient record
  - runs `check_drug_interactions(...)` with a 10 second timeout
  - scans prior assistant messages for imaging context containing
    `Imaging Findings`
  - passes patient context, imaging context, current medications, and drug alerts
    into `generate_soap_note(...)`

Expected backend behavior:

- The SOAP endpoint should produce richer notes because it now has medication,
  interaction, patient, and imaging context.
- If drug interaction checks fail or time out, SOAP generation should continue
  with empty drug alerts.

### Vision Tooling: `src/vision_tool.py`

Commit `72b2465` updated Vertex MedGemma authentication and routing:

- Added `subprocess` import.
- Added `_gcloud_vertex_token()` to get an access token from the active `gcloud`
  account.
- Added `_adc_vertex_token()` to get a token from service-account credentials or
  Application Default Credentials.
- Updated `call_medgemma_vertex(...)` to:
  - try ADC/service-account auth first
  - fall back to `gcloud auth print-access-token` when ADC fails and no explicit
    service account JSON is configured
  - retry with a `gcloud` token on 401/403 responses when using ADC
- Updated OpenRouter fallback model ordering so a preferred model beginning with
  `vertex/` is not inserted into the OpenRouter vision fallback list.

Expected vision behavior:

- Local developers who have `gcloud` authenticated can still use Vertex
  MedGemma when ADC is not configured correctly.
- Explicit service-account JSON failures still surface instead of being silently
  replaced by user-token auth.
